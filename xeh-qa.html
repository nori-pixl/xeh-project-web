よくある質問 (Q&A)

xehの設計・使い方・トラブルシューティングに関する疑問にお答えします

xeh は「XMLベースのマルチランタイム・オーケストレーションエンジン」です。Python・Go・Ruby・Rustなど複数の言語ランタイムを、1つの設定ファイル群（set.json と app.xeh）で束ねて並列実行できます。

解決する主な課題は「ポリグロット環境の複雑さ」です。通常、複数言語を連携させるにはメッセージキューやマイクロサービス基盤が必要ですが、xehでは親プロセス（Go）が共有メモリをJSONとして各子プロセスのstdinへ直接流し込むため、追加インフラなしで変数共有が実現します。

xehエンジン本体のビルドには Go 1.18以降 が必要です。それ以外に必要なのは、実際に使う各言語のランタイムだけです。

例えば <py-processor> タグを使うなら Python が、<rb-script> を使うなら Ruby がインストールされている必要があります。使わない言語のランタイムは不要です。

set.json は「環境定義」です。どの言語タグがどのコマンドで実行されるかをマッピングします。インフラの設定書と考えてください。

app.xeh は「アプリケーション定義」です。共有メモリの初期値（変数）と、各サブランタイムで動かす実際のコードロジックを記述します。

この分離により、同じ app.xeh を本番・開発・テスト環境で使い回しつつ、set.json だけ差し替えてコマンドパスを変えるといった運用が可能です。

set.json の "runtimes" セクションにエントリを追加し、"engines" で対応するタグ名を定義するだけです：

{
  "runtimes": {
    "deno": {
      "command": "deno",
      "args": ["run", "--allow-read", "{src}"]
    }
  },
  "engines": {
    "deno-worker": {
      "type": "deno",
      "src": "src/worker.ts"
    }
  }
}

これで app.xeh 内で <deno-worker>...</deno-worker> タグが使えるようになります。

xehエンジンは起動時、app.xeh の該当タグの中身を engines[tag].src で指定したパスのファイルとして一時書き出します。その後 args の {src} をそのパスで置換してコマンドを組み立てます。

例えば "args": ["-u", "{src}"] は実際には python -u src/process.py というコマンドに展開されます。args 配列の任意の位置に複数回記述することもできます。

app.xeh に <server> タグを追加します。id で名前を、port でポート番号を指定し、内部にHTMLを直接記述できます：

<xeh>
  <server id="web-ui" port="3000">
    <!DOCTYPE html>
    <html><body>
      <h1>xeh Web UI</h1>
    </body></html>
  </server>
</xeh>

複数の <server> タグを並べることで、異なるポートで複数のサーバーを同時に立ち上げることができます。/submit エンドポイントではフォームデータの受け取りと共有メモリの表示が自動で処理されます。

xehエンジンは子プロセス起動後、MemoryPacket 構造体をJSONにシリアライズし、子プロセスのstdinへ1行で書き込みます：

{"memory_name":"production_space","data":{"threshold":85,"status":"active"}}

子プロセス側では起動直後にstdinの最初の1行を読み取るだけで共有変数にアクセスできます（Python例）：

import sys, json
packet = json.loads(sys.stdin.readline())
threshold = packet["data"]["threshold"]

これはOS標準のパイプを使った方式のため、対応言語を問わずどのランタイムでも同じパターンで受け取れます。

xehエンジンは内部で3種類のMutexを使ってRace Conditionを防いでいます：

• storeMu (sync.RWMutex)：共有データストアへの読み書きを保護。複数goroutineが同時に読めるが、書き込みは排他的に行われます。

• processMu (sync.Mutex)：実行中プロセスのリスト追加/参照を保護します。

• serverMu (sync.Mutex)：HTTPサーバーインスタンスのリスト管理を保護します。

また、各goroutineのクロージャ変数は引数として明示的にコピーされており、ループ変数キャプチャのバグも防止されています。

現バージョン（v1.0.0）では、データフローは 親→子の一方向 です。子プロセスからのデータを親の共有ストアに書き戻す仕組みはまだ組み込まれていません。

現時点での回避策としては、子プロセスがファイルやローカルソケットに結果を書き出し、次のプロセスや <xeh-logic> でそれを読み込む設計パターンが有効です。双方向通信は将来のロードマップで検討されている機能です。

まず set.json の "plugins" セクションに外部xehファイルへのパスを登録します：

{
  "plugins": {
    "ai-mod": { "src": "plugins/ai_plugin.xeh" }
  }
}

次に app.xeh の <subframework> タグで import 属性を指定します：

<subframework import="ai-mod">
  <!-- このsubframeworkのノードが実行される前にai_plugin.xehがロードされます -->
</subframework>

インポートされた .xeh ファイルの <root> に定義された変数は、メインの sharedStore にマージされます。同名の変数はインポート側で上書きされます。

現バージョンではネイティブXMLロジックとして実装されているのは <print> のみです。value 属性または テキストコンテンツに ${変数名} を記述すると sharedStore の値が展開されます。

ただし <xeh-logic> のノード解析自体は汎用的に設計されており、エンジン本体（main.go）の該当箇所に条件分岐を追加することで <if> <set> などのカスタムタグを比較的容易に実装できます。

最も多い原因は set.json の "engines" セクションのタグ名と app.xeh のXMLタグ名の不一致です。以下を確認してください：

• set.json の engines キー名と app.xeh のタグ名が完全に一致しているか（大文字小文字・ハイフン含む）

• engines[tag].type の値が runtimes に存在するキー名と一致しているか

• 指定したランタイムコマンド（例: python3）がシステムのPATHで見つかるか → ターミナルで直接コマンドを実行して確認

これは現バージョンの既知の制約です。app.xeh のタグ内に埋め込むコードはGoの標準 encoding/xml パーサーで処理されるため、XMLの特殊文字をそのまま書けません。

回避策として、コードブロックを <![CDATA[...]]> セクションで囲む方法が有効です：

<py-processor><![CDATA[
if threshold < 100:
    print(f"値: {threshold}")
]]></py-processor>

あるいは、コードを外部ファイルに分離して engines[tag].src でそのパスを直接指定する設計も検討してください。

xehはLinux/macOSでは sh -c 、Windowsでは cmd /C でコマンドを実行します。この場合、シグナルハンドラが Kill() するのはシェルプロセス自体であり、その子（実際のインタプリタ）が残るケースがあります。

根本的な解決策としては、exec.Command でシェルを介さず直接コマンドを起動する設計変更、または syscall.SysProcAttr{Setpgid: true} でプロセスグループを設定して syscall.Kill(-pgid, syscall.SIGKILL) でグループごとkillする方法が有効です。
